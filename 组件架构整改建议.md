# 组件架构整改建议

## 背景

当前 `src/components` 已经具备基础分层：

- `components/ui`：底层 UI primitive。
- `components/admin`：后台管理通用组件。
- `components/dashboard`：后台工作台展示组件。
- `components/brand`：品牌视觉组件。

整体结构可以支撑当前阶段开发，但如果目标是达到更接近大厂长期维护标准，需要进一步明确组件归属、业务边界和依赖方向，避免后续业务扩展后出现组件目录膨胀、页面过厚、通用组件夹带业务语义等问题。

## 当前主要问题

### 1. 页面职责偏重

`src/routes/admin/users.tsx` 同时承担了以下职责：

- 路由页面声明。
- 筛选状态管理。
- 分页状态管理。
- 编辑弹层状态管理。
- 批量操作状态管理。
- 表格列定义。
- 用户状态切换、删除、批量禁用等行为编排。
- 具体表单 UI。

这会导致页面文件越来越厚，也会让用户模块的 UI 逻辑难以复用和测试。

### 2. 后台 CRUD 模式组件归属不够精准

`ResourcePage`、`ResourceTable`、`FilterBar`、`QueryFormItem` 本质上不是普通后台布局组件，而是后台资源管理页面的复用模式。

当前放在：

```txt
src/components/admin/layout
src/components/admin/data-table
```

更准确的归属应该是后台资源管理 pattern，例如：

```txt
src/patterns/admin-resource
```

### 3. `dashboard` 顶层目录边界不稳定

`src/components/dashboard` 当前只服务 `/admin` 工作台页面。按长期演进标准，顶层 `components` 不应承载具体页面主题组件。

如果这些组件只服务后台首页，应移动到：

```txt
src/features/admin-dashboard/components
```

或路由局部组件目录：

```txt
src/routes/admin/-components
```

### 4. `brand` 目录需要明确定位

`TechHero`、`GlowCube` 如果是全站品牌视觉组件，可以继续留在 `components/brand`。

如果它们只服务后台工作台，应跟随后台首页业务移动到：

```txt
src/features/admin-dashboard/components
```

### 5. 组件依赖规则尚未显式建立

当前还没有明确约束不同层级之间的依赖关系。随着业务增加，容易出现：

- 底层 UI 反向依赖业务。
- 通用组件引用具体 feature。
- feature 之间互相随意 import。
- route 页面沉淀大量业务 UI。

## 分层职责

### `components/ui`

只放无业务语义的基础 UI primitive。

允许：

- Button
- Dialog
- Sheet
- Select
- Input
- Table
- Checkbox
- Tooltip

禁止：

- 引用 `features/*`
- 引用 `routes/*`
- 出现用户、角色、菜单、权限等业务语义
- 包含接口请求或状态管理

### `components/common`

放跨业务、跨页面的轻业务通用展示组件。

适合放：

- `EmptyState`
- `StatusBadge`
- `ErrorMessage`
- `PageLoading`

不适合放：

- `UserAvatar`
- `RoleSelect`
- `MenuTree`

这些应归入具体 feature。

### `layouts`

放应用级布局。

适合放：

- `AdminShell`
- `AdminHeader`
- `AdminSidebar`
- `AdminBreadcrumb`

布局组件可以消费通用 hooks 和菜单查询，但不要沉淀具体业务模块的 UI。

### `patterns`

放跨业务复用的交互模式。

后台系统中典型 pattern：

- 资源管理页
- 查询表单
- 数据表格
- 分页
- 批量操作入口
- 表单弹层
- 确认操作

`patterns` 可以依赖 `components/ui` 和 `components/common`，但不能依赖具体 `features/users`、`features/roles` 等业务模块。

### `features`

放业务域内聚代码。

每个业务域应包含：

- 业务类型
- schema
- actions
- queries
- mutations
- policy
- service
- 业务专属组件
- 业务专属 hooks

例如用户模块：

```txt
src/features/users/
  components/
    user-management-view.tsx
    user-filters.tsx
    user-table.tsx
    user-edit-form.tsx
    user-avatar.tsx
  hooks/
    use-users-page-controller.ts
  users.actions.ts
  users.queries.ts
  users.schema.ts
  users.types.ts
```

### `routes`

路由文件只做装配。

理想状态下，`routes/admin/users.tsx` 应接近：

```tsx
import { createFileRoute } from "@tanstack/react-router";

import { UserManagementView } from "~/features/users/components/user-management-view";
import { useUsersPageController } from "~/features/users/hooks/use-users-page-controller";

export const Route = createFileRoute("/admin/users")({
  component: AdminUsersPage,
});

function AdminUsersPage() {
  const controller = useUsersPageController();

  return <UserManagementView controller={controller} />;
}
```

## 依赖规则

建议建立以下单向依赖规则：

```txt
routes
  -> features
  -> patterns
  -> components/common
  -> components/ui
```

允许：

```txt
routes/admin/users.tsx -> features/users
features/users -> patterns/admin-resource
features/users -> components/common
features/users -> components/ui
patterns/admin-resource -> components/common
patterns/admin-resource -> components/ui
layouts/admin -> components/ui
```

禁止：

```txt
components/ui -> features/*
components/common -> features/*
patterns/* -> features/*
features/users -> routes/*
features/users -> features/roles
features/roles -> features/users
```

如确实存在跨 feature 共享逻辑，应抽到：

```txt
src/lib
src/components/common
src/patterns
```

## 推荐迁移步骤

### 第一阶段：低风险搬迁

1. 新建 `src/patterns/admin-resource`。
2. 移动以下组件：
   - `ResourcePage`
   - `ResourceTable`
   - `FilterBar`
   - `QueryFormItem`
3. 更新 import 路径。
4. 确认用户管理页功能不变。

目标：先把后台 CRUD pattern 从普通 components 中剥离出来。

### 第二阶段：拆薄用户页面

1. 新建：

```txt
src/features/users/components
src/features/users/hooks
```

2. 从 `routes/admin/users.tsx` 拆出：

```txt
user-management-view.tsx
user-filters.tsx
user-table.tsx
user-edit-form.tsx
user-table-columns.tsx
use-users-page-controller.ts
```

3. 让 `routes/admin/users.tsx` 只保留路由声明和页面装配。

目标：把用户业务 UI 收敛回用户 feature。

### 第三阶段：整理后台首页组件

1. 新建：

```txt
src/features/admin-dashboard/components
```

2. 移动：

```txt
components/dashboard/*
components/brand/tech-hero.tsx
components/brand/glow-cube.tsx
```

如果后续确认 `brand` 组件会被登录页、官网页、后台页共同复用，再保留在 `components/brand`。

目标：减少顶层 `components` 中的页面主题组件。

### 第四阶段：布局组件独立

1. 新建：

```txt
src/layouts/admin
```

2. 移动：

```txt
components/admin/layout/admin-shell.tsx
components/admin/layout/admin-header.tsx
components/admin/layout/admin-sidebar.tsx
components/admin/layout/page-header.tsx
components/admin/layout/use-sidebar-collapsed.ts
components/admin/layout/use-nav-state.ts
components/admin/layout/admin-nav-config.ts
components/admin/layout/menu-icons.ts
```

3. 如果 `PageHeader` 不是 admin 专属，可考虑放到 `components/common/page-header.tsx`。

目标：应用布局和业务组件解耦。

### 第五阶段：建立架构约束

建议补充 lint 规则或约定文档，明确禁止反向依赖。

可选方案：

- 使用 ESLint `no-restricted-imports`。
- 使用 dependency-cruiser 做依赖图检查。
- 在 PR checklist 中增加组件归属检查项。

## 迁移后的验收标准

达到 10 分结构时，应满足：

- `components/ui` 不包含任何业务语义。
- `components/common` 不依赖 `features`。
- `patterns` 不依赖具体业务模块。
- `features/users` 拥有用户业务自己的组件和 hooks。
- `routes/admin/users.tsx` 文件明显变薄，只负责装配。
- 后台首页组件不再散落在顶层 `components/dashboard`。
- 布局组件不再混在业务组件目录中。
- 新增一个业务模块时，有明确位置放置页面、组件、接口、schema、types、hooks。
- 新增一个通用组件时，能明确判断应该进入 `ui`、`common`、`patterns` 还是某个 `feature`。

## 建议优先级

优先级从高到低：

1. 拆薄 `routes/admin/users.tsx`。
2. 将 `ResourcePage`、`ResourceTable`、`FilterBar` 移入 `patterns/admin-resource`。
3. 将 `components/dashboard` 收敛到 `features/admin-dashboard/components`。
4. 将 `components/admin/layout` 移入 `layouts/admin`。
5. 建立 import 边界规则。

## 总结

当前结构可以继续开发，但还属于“中小型后台项目的良好分层”。如果目标是大厂级长期维护标准，核心不是增加目录，而是建立清晰边界：

- 基础组件归基础层。
- 复用交互模式归 pattern。
- 业务 UI 归 feature。
- 应用布局归 layout。
- 路由只做装配。

按上述步骤迁移后，组件体系会更容易扩展、复用、测试和审查，也能显著降低后续业务增长带来的架构腐化风险。
